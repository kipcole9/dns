# Phase 5+ — recursion, clustering, policy

Three new strands requested 2026-05-02 after Phase 4 wrap.

## 9. Recursive resolution

Server can recurse on RD=1 queries when we don't hold the zone.

- [ ] Root hints (13 root servers, A + AAAA glue)
- [ ] Cache (ETS-backed, TTL-aware) under `ExDns.Cache`
- [ ] Outbound resolver client (`ExDns.Recursor.Client`) — UDP first, TCP retry on TC, EDNS0 advertised
- [ ] Iterative resolver (`ExDns.Recursor.Iterator`):
  - [ ] start at the closest cached delegation, fall back to root
  - [ ] follow NS delegations
  - [ ] follow CNAME chains across zones
  - [ ] honour query-name minimisation (RFC 9156) — defer if too invasive for Phase 5
  - [ ] max-depth and max-time guards
- [ ] Hybrid resolver (`ExDns.Resolver.Hybrid`) that tries `Resolver.Default` first, falls back to recursion when RD=1 and no zone matches
- [ ] Configuration: `:resolver_module` switches between Default / Hybrid; `:recursion_enabled` and `:cache_max_entries` knobs
- [ ] Tests: against a fixture upstream + dig integration via the live network (skippable tag)
- [ ] DNSSEC validation: explicitly out of scope

## 10. Cluster support

≥ 3 nodes serve the same zones. Exactly one node is the update master at any time, dynamically elected. Updates commit transactionally across the cluster.

- [ ] Pluggable storage behaviour (`ExDns.Storage` behaviour) so we can swap ETS for the cluster backend
- [ ] `ExDns.Storage.Mnesia` backend — disc_copies / ram_copies on every node, distributed transactions
- [ ] `ExDns.Cluster` supervisor that:
  - [ ] connects to peers via static `:cluster_nodes` list (works for tests, no deps)
  - [ ] participates in master election via `:global.register_name(:ex_dns_update_master, self())`
  - [ ] re-runs election on `nodedown` / `:DOWN`
  - [ ] exposes `master?/0`, `master_node/0`, `nodes/0`
  - [ ] optionally consumes libcluster topologies for production discovery
  - [ ] partisan flagged as the swap when `:global`/BEAM-dist isn't enough (replaces distribution layer entirely)
- [ ] All write paths go through `ExDns.Cluster.Update.commit/1` which:
  - [ ] runs locally on master, returns `{:committed, version}` after the Mnesia transaction returns `:atomic`
  - [ ] forwards to master from non-master nodes via `GenServer.call`
- [ ] Read paths stay local (every node holds the full state)
- [ ] Tests:
  - [ ] start three local nodes (`Node.start` + `:net_kernel.start`)
  - [ ] put a zone on the master, assert all three see it via lookup
  - [ ] kill the master, assert a new one is elected within ~1s
  - [ ] verify forwarded writes from a non-master node land everywhere

## 11. Policy-based resolution

Admin configures a chain of policies that may halt resolution early with a synthesised response, based on properties of the inbound request.

- [ ] `ExDns.Request` struct: `%{message, source_ip, source_port, transport, received_at}`
- [ ] Listener changes:
  - [ ] UDP, TCP, DoH listeners construct an `ExDns.Request` and pass it to the resolver
- [ ] Resolver behaviour upgrade:
  - [ ] `@callback resolve(ExDns.Request.t() | ExDns.Message.t()) :: ExDns.Message.t()` (back-compat: still accepts a Message)
- [ ] `ExDns.Policy` behaviour:
  - [ ] `@callback init(opts) :: state`
  - [ ] `@callback resolve(request, state) :: {:halt, response} | :continue`
- [ ] `ExDns.Resolver.Policy` chain runner: walks a configured list of `{module, opts}`, returns the first `:halt` response, otherwise delegates to the underlying resolver
- [ ] Built-in `ExDns.Policy.SourceIp` — maps source CIDR → answer transform (e.g. "rewrite the A record to a region-specific IP")
- [ ] Configuration:
  ```
  config :ex_dns,
    resolver_module: ExDns.Resolver.Policy,
    policies: [
      {ExDns.Policy.SourceIp, table: %{
        {{198,51,100,0},24} => %{a: {192,0,2,1}},
        {{203,0,113,0},24}  => %{a: {192,0,2,2}}
      }}
    ]
  ```
- [ ] Tests:
  - [ ] unit tests against the policy chain with a stubbed underlying resolver
  - [ ] dig integration that sends queries from two source IPs (via local-bind) and asserts different answers

## Cross-cutting

- [ ] When clustering is enabled, the policy chain must run identically on every node (same config)
- [ ] When recursion is enabled and the cluster has a cache, cache writes should NOT replicate (each node has its own cache; replication would thrash on TTL skew)

# Phase 6 — mDNS (RFC 6762)

Multicast DNS responder for the `.local` namespace.

## 12. mDNS responder

- [ ] `ExDns.Listener.MDNS` — UDP listener bound to 224.0.0.251:5353
  (IPv4) and FF02::FB:5353 (IPv6) with `SO_REUSEADDR` +
  `SO_REUSEPORT` so it co-exists with the OS's mDNSResponder on macOS
- [ ] Multicast group join via `{:add_membership, {group, iface}}`
- [ ] `ExDns.Resolver.MDNS` — variant resolver that only answers for
  names ending in `.local`; ignores RD; supports the QU (top bit of
  QCLASS) "unicast response requested" flag
- [ ] Response routing: QU set → unicast back to the querier; QU
  cleared → multicast to the group
- [ ] TTL semantics: `TTL=0` is "goodbye" — accept and treat as cache
  flush
- [ ] Storage: re-use `ExDns.Storage` with apex `"local"` for
  `.local` records
- [ ] Tests:
  - [ ] unit: QU bit decode, multicast vs unicast routing
  - [ ] integration: send a query via `:gen_udp` to 224.0.0.251:5353,
    assert our response arrives (skippable on hosts where the OS
    mDNSResponder is too aggressive)

## 13. mDNS announce + probe (RFC 6762 §8) — follow-up

- [ ] Probe sequence: 3 queries 250 ms apart for the names we plan to
  advertise; if any reply, we have a conflict
- [ ] Announce: 2 unsolicited responses 1 s apart
- [ ] Conflict resolution: rename (`-2` suffix) or refuse
- [ ] Triggered when zone for `.local` is loaded
